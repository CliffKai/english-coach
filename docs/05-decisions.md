# 决策记录（ADR）

记录关键设计决策及其理由，避免后续反复纠结。

---

## ADR-001：Web 应用 + 第一版含语音

- **决策**：第一版做 Web 应用，且包含语音功能（口语练习/对话）。
- **理由**：Web 天然支持录音/播放、富文本批改高亮；语音是口语练习的核心体验，第一版不能省。
- **代价**：开发量大于 CLI，需处理 STT/TTS 管线与 WebSocket 流式。

## ADR-002：一切外部依赖皆适配器（存储 + 模型 + 语音）

- **决策**：存储、LLM、STT、TTS、发音评估全部用「接口 + 适配器」抽象。
- **理由**：用户要求支持云服务商模型与自部署模型；存储要可在本地/云端间选择。统一成适配器模式，业务代码零耦合。
- **落地**：LLM 重点做 `OpenAICompatAdapter`（覆盖 80% 模型）+ `ClaudeAdapter`（原生，做评分）。

## ADR-003：发音评估不自研，留适配器，默认关闭

- **决策**：不自研发音评估；做成 `PronunciationProvider` 适配器，第一版默认 `NoneAdapter`。
- **理由**：发音评估是独立的声学难题，自研等于重做声学模型，投入产出极差；成熟方案（Azure Pronunciation Assessment、讯飞、Kaldi GOP）已很好。
- **第一版行为**：口语打分中发音(pronunciation)、流利度(fluency)维度标注「基于文本估算，仅供参考」，其余维度（内容/词汇/语法/连贯）正常评分。用户填 Azure key 即升级为音素级评分。

## ADR-004：生词只存来源句，不存释义

- **决策**：`VocabEntry` 存 `word + lemma + context_sentences[]`，**不存任何释义文字**。
- **理由**：
  - 只存单词会丢失语境，回头不记得为何不认识，违背"理解式记忆"。
  - 存释义会：① 多一次 LLM 成本；② 制造"标准答案"，用户去比对 → 滑回死记硬背；③ 多义词存哪个义项是麻烦。
  - 只存来源句：空间极省（句子本就在手）、语境天然消歧（"sat by the river bank"自锁定河岸义）、完美契合功能3a（意思由用户当场理解出来）。
- **一句话**：意思不该被存储，意思应在复习时被重新理解出来；来源句是钥匙。
- **多义处理**：同词不同义 → 存多条 `context_sentences`，背诵时一起呈现，建立立体认知。

## ADR-005：考试模式零脚手架，但允许提前交卷

- **决策**：考试模式（free_write / dialogue）不救场、不给提示，但用户可随时「提前交卷」，按已有内容打分+复盘。
- **理由**：压力是刻意设计，防止用户形成依赖；提前交卷不是救场（不给词/提示），只是允许早结束，符合真实考试。
- **附带**：卡顿/停顿作为流利度评分信号记录进复盘，不抹平。

## ADR-006：模型按任务分配

- **决策**：评分用最强模型，切词等高频轻任务用本地/轻量模型，对话用性价比模型。用户可配置。
- **理由**：兼顾评分准确性与高频任务成本。

## ADR-007：无账号系统，单用户本地优先，schema 预留 user_id

- **决策**：不做注册/登录；默认单用户（`user_id = "local-user"`），本地优先。但数据模型保留 `user_id` 字段。
- **理由**：开源自部署工具的典型用户是 clone 下来本地自用（参考 Anki 等）。强加账号是负担，劝退使用者与贡献者。保留 `user_id` 字段则让未来 fork 成多用户 SaaS 只需加一层认证，零成本保留可能性。

## ADR-008：后端语言定 Python (FastAPI)

- **决策**：后端用 Python + FastAPI；前端 React/TS。
- **理由**：核心依赖几乎全在 Python 生态 —— spaCy（切词/词元还原）、faster-whisper（本地 STT）、FSRS（官方库）。用 Node 会被迫起 Python 子进程做切词和语音，变成双语言维护。不为"前后端同语言"牺牲整个 ML 生态。

## ADR-009：开源落地基线纳入 MVP

- **决策**：数据导入/导出、首次配置向导、一键启动（docker-compose）、「今日学习」首页 一并纳入 MVP，而非阶段2。
- **理由**：开源项目"能不能被人跑起来、能不能形成日常习惯"由这几项直接决定。
  - 导入/导出（尤其生词本导出 Anki）：本地优先用户最怕数据被锁死；Anki 兼容是引流与留存点。
  - 配置向导：落地第一道坎是配置 API key 与模型，无向导则新用户跑不起来。
  - 「今日学习」首页：三大功能是平行的，需要一个聚合页把碎片串成日常习惯，否则用户不知从何下手。

## ADR-010：多义词调度单位 = 按词（per-lemma），不按义项

- **决策**：FSRS 调度以 `VocabEntry`（即一个 lemma）为单位，一个生词条目持有**唯一** `fsrs_state`；同词不同义共用同一调度节奏。多义靠 `context_sentences` 多条并呈消解，不拆成多个义项各自调度。
- **背景**：这是 `07-implementation-order.md` 列出的红线——「一个 VocabEntry 存多条 context_sentences（多义）却只有一个 fsrs_state，按词还是按义项调度，需在 L2 接 FSRS 前定下」。L2 接入 FSRS 前在此定案。
- **理由**：
  - schema 已对 `(user_id, lemma)` 建唯一索引、每个 `VocabEntry` 仅一个 `fsrs_state`——按义项需为每个义项存独立调度状态，与既有结构冲突。
  - 按义项必须为每个义项贴标签/释义以区分，等于变相存释义，违背 ADR-004（不存释义）。
  - 理解式记忆（功能3a）本就把多义来源句**一起**呈现以建立「立体认知」；同词多义一起复习正是该设计，不该被拆散。
  - 简单可解释：用户看到的是「这个词到期了」，而非「这个词的第2个义项到期了」。
- **代价**：某词一个义项熟、另一义项生时，调度无法分别对待（整体一个节奏）。可接受——多义来源句并呈时熟义项顺带复习成本极低；真有强需求留待阶段2的「义项级」扩展（需新 ADR + schema 迁移）。
- **关联**：ADR-004（只存来源句不存释义）；落地见 `app.scheduling.fsrs_scheduler` 与 `FsrsState`。

## ADR-011：F3a「模糊理解判断 → 离散 FSRS 评级」的映射

- **决策**：理解式背词（F3a）是开放式判断（用户用自己的话说理解，LLM 判断是否到位），但 FSRS 要离散评级（again/hard/good/easy）。定义如下映射，落地在 `MemoryWordAgent`：
  - LLM 只输出三档 verdict：`correct`（抓住该语境下的意思）/ `partial`（沾边但不准、漏义或含糊）/ `wrong`（理解错或答非所问）；再加一个用户自评的 `too_easy`（秒答、太简单）旁路。
  - 映射：`too_easy → EASY`（优先级最高，短路不调 LLM）；`correct → GOOD`；`partial → HARD`；`wrong / 空答 → AGAIN`（空答短路不调 LLM）。
  - 解析失败或非法 verdict → 保守归 `partial`（HARD），宁可多复习一次。
  - 判断以**复习卡上展示的全部来源句**为语境（多义并呈，ADR-010），不只取第一句——否则用户解释了后面的句子会被误判。
- **毕业（learning → known）门槛**：用**连续**答好次数而非累计复习次数——`连续 GOOD/EASY ≥ 3` 才标 known（退出 due 队列）。累计会让 `again,again,good` 这种「近期才答对」的词误毕业。连续计数存于 `FsrsState.consecutive_good`（项目自有字段，非 fsrs 原生），由调度器在每次 `review` 时维护：答好累加、答差清零。已 known 的词若又收集到新义项/被标不认识，由 F1 拉回 learning 重新进队列（见 `TokenizerAgent.collect`）。
- **背景**：这是 `07-implementation-order.md` 列出的红线——「F3a 是开放式『说出理解』由 LLM 判断，但 FSRS 要离散评级，这个『模糊判断 → 离散评级』的映射必须先定义清楚，否则调度全错」。L3 接 F3a 时在此定案。
- **理由**：
  - 让 LLM 直接输出 4 档评级会把「评估理解」与「决定复习节奏」两件事耦合进一次模糊判断，方差大、不可控；拆成「LLM 只判理解对错粗细 + 确定性映射到评级」更稳更可解释。
  - 三档（对/沾边/错）粒度足够区分调度强度，又不逼 LLM 做它不擅长的细分；`too_easy` 交给用户自评（FSRS 的 Easy 本就该由「主观觉得太简单」触发）。
  - 判分只看「是否抓住该语境的意思」，不苛求措辞——契合 ADR-004「理解式、不比对标准释义」。
- **关联**：ADR-004（只存来源句不存释义）、ADR-010（按词调度）；评级枚举见 `app.scheduling.ReviewRating`。

## ADR-012：语音（STT/TTS）也走 OpenAI 兼容协议为默认 + 本地适配器可选，用户自配

- **决策**：`STTProvider` / `TTSProvider` 沿用 LLM 的「一个 OpenAI 兼容适配器打天下」思路：
  - **STT 默认 `OpenAICompatSTTAdapter`**：打 OpenAI 音频协议 `/v1/audio/transcriptions`，填 `base_url + api_key + model` 即可，云端（OpenAI / Groq…）或本地服务（faster-whisper-server、`openedai-whisper` 等暴露同协议者）皆可。
  - **TTS 默认 `OpenAICompatTTSAdapter`**：打 `/v1/audio/speech`，同上。
  - 各保留一个**纯本地可选适配器**：`FasterWhisperSTTAdapter`（faster-whisper，离线）与一个本地 TTS 适配器，给「完全离线/隐私优先」用户。**默认走 API，本地为可选**（B3）。
- **背景**：`docs/04` 原写死 STT=`WhisperLocalAdapter(faster-whisper)`、TTS=`本地 TTS`。L4 接入前改为与 LLM 适配器一致的「协议优先、云/本地自配」形态。
- **理由**：
  - OpenAI 的 `/v1/audio/transcriptions`（STT）与 `/v1/audio/speech`（TTS）已是事实标准，大量云/自部署服务照它实现——做好一个兼容适配器，填不同 `base_url` 即覆盖绝大多数，复刻 `OpenAICompatAdapter`（LLM）已验证的核心洞察。
  - 不把第一版绑死在 faster-whisper 的重依赖上（ctranslate2 等），让「想用云就填 key、想离线就指本地服务」二者皆可，降低开源用户上手门槛。
  - 语音连接信息（`base_url/api_key/model/kind`）与 LLM 一样**只来自环境变量、绝不入库**，复用 `AppConfig` 的嵌套配置形态（新增 `stt_providers` / `tts_providers`，镜像 `llm_providers`）。
- **重依赖处理**：faster-whisper 等放 `[voice]` 可选依赖组，适配器内**懒加载**（沿用 `SpacyTokenizer`/`FsrsScheduler` 的 import-inside 风格），模块导入仍轻、测试全程离线 mock，不污染环境也不阻塞构建。
- **代价**：纯本地默认体验需用户自行起一个本地语音服务或装 `[voice]`；可接受——与「本地优先但不牺牲易用」的整体取向一致，ADR-002 的可换适配器保证随时切换。
- **关联**：ADR-001（第一版含语音）、ADR-002（一切外部依赖皆适配器）、ADR-003（发音评估独立适配器，默认 None）、ADR-006（按任务分配模型，语音同理按 provider 名查连接）。

## ADR-013：口语流利度/发音——接发音评估 API 才有真分，否则两维空缺并标注

- **决策**：口语打分（F2b/F2d）的**发音(pronunciation)与流利度(fluency)** 两维，由 `PronunciationProvider` 决定：
  - 默认 `NoneAdapter`（ADR-003）：这两维**不给分数**（分值留空 / `null`），结果标 `estimated=True`，前端明示「未接入发音评估，无此项评分」。**不**用文本/转写去假装能评发音。
  - 用户**配置了发音评估 API（如 Azure Pronunciation Assessment）→ 自动得到音素级真实评分**，`estimated=False`。
  - 其余维度（内容/词汇/语法/连贯）始终由 STT 转写文字正常评分，不受发音 API 有无影响。
- **背景**：把用户语音经 STT 转成文字再打分，会丢失声学信息——发音准不准、是否地道，文字看不出。设计上不掩盖这一局限，而是诚实地「有 API 才评、没有就空着」。
- **理由**：
  - 发音是声学问题，文字零信息；与其糊弄一个不可信的发音分，不如空缺并标注，避免误导用户（延续 ADR-003「标注估算/仅供参考」的诚实原则，并更进一步：发音维度宁可空缺也不假评）。
  - 真发音评分不自研、接成熟 API（ADR-003），用户「配了就生效」是最小心智负担。
- **与 ADR-003 的关系**：ADR-003 定「发音评估留适配器、默认 None、标注估算」；本 ADR 明确**默认 None 时发音/流利度两维取『空缺 + 标注』而非『文本估算一个数』**，并定下「配 API 即真分」的开关语义。STT 转写的**时间戳**（`Transcript.segments`）仍可作流利度的辅助信号记入复盘（ADR-005「停顿是流利度信号」），但不据此编造流利度分数。
- **关联**：ADR-003、ADR-005、ADR-012；落地见 `PronunciationProvider` 与 `ExaminerAgent` 口语打分分支。

## ADR-014：Anki 导出——卡背放「来源句 + 用户理解历史」，不放释义；CSV 先行

- **决策**：生词本导出 Anki 时，卡片**正面 = 单词（word）**，**卡背 = 该词的全部来源句（`context_sentences`）+ 用户历次复习说出的理解（`user_understanding`）**，**不生成、不存储任何「标准释义」**。第一版导出格式为 **CSV**（Anki「文件→导入」原生支持）；`.apkg` 留作阶段2增强。
- **背景**：这是 `07-implementation-order.md` 点名的 L5 硬骨头——「ADR-004 不存释义，但 Anki 卡片要正反面，卡背放什么？」。导出前必须定案，否则要么违背 ADR-004（卡背塞释义），要么导出一张只有正面的废卡。
- **理由**：
  - **忠于 ADR-004（不存释义）**：卡背放的是「钥匙」（来源句）+「用户自己的话」（理解历史），二者都已在库，导出**零新增存储、零 LLM 调用、全程离线**。卡背不是「标准答案」，用户复习时仍是「看句子→当场重新理解」，而非比对一条权威释义——与功能3a「理解式记忆」的设计一脉相承。
  - **多义天然呈现**：同词多义存多条来源句（ADR-004/010），导出时一并进卡背，Anki 上复习即「多义并呈」，无需拆卡。
  - **用户理解历史是真实资产**：`user_understanding` 是用户亲自复述过的理解（非机器释义），放卡背既是「自己写给自己的提示」，又不构成外部标准答案；空（从未复习）则卡背只有来源句，仍可用。
  - **CSV 先行**：纯文本、零新增依赖、Anki 原生导入、测试可纯离线断言文本，契合本项目「测试全程离线」的约束。`.apkg`（zip 包，需引入 genanki + 自带 note type 模板）体验更好但增加依赖与二进制测试成本，收益不紧迫，归阶段2。
- **导出形态（CSV）**：两列 `Front,Back`（UTF-8，标准 CSV 转义换行/逗号/引号）。卡背用换行分隔来源句区与理解区，例：
  - Front：`ephemeral`
  - Back：`【来源句】\n• Ephemeral apps are a serendipitous trend.\n• The beauty was ephemeral.\n【我的理解】\n• 短暂的、转瞬即逝的`
  - 导入 Anki 时选「字段由逗号分隔」「允许字段内 HTML/换行」即可；卡背无理解记录时仅含来源句区。
- **与 JSON 全量导出的分工**：JSON 全量备份（ADR-009「导入/导出」）是**无损迁移**用途（含 `fsrs_state`/`status`/时间戳等全字段，可再导回本应用）；Anki CSV 是**对接外部 Anki 工作流**用途（只取 word + 卡背文本，FSRS 调度交给 Anki 自身，不回传）。两者目的不同，并存。
- **代价**：Anki 卡没有「权威释义」，习惯了正面单词背面中文释义的用户初看会不适应——但这正是本产品的刻意取向（ADR-004），导出不应背叛它。真需要释义的用户可在 Anki 内自行补 `.apkg` 阶段2再议。
- **关联**：ADR-004（只存来源句不存释义）、ADR-009（导入导出纳入 MVP）、ADR-010（按词调度/多义并呈）；落地见 `app/api/data.py` 的 Anki 导出分支。
