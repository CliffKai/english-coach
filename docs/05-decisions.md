# 决策记录（ADR）

记录关键设计决策及其理由，避免后续反复纠结。

---

## ADR-001：Web 应用 + 第一版含语音

- **决策**：第一版做 Web 应用，且包含语音功能（口语练习/对话）。
- **理由**：Web 天然支持录音/播放、富文本批改高亮；语音是口语练习的核心体验，第一版不能省。
- **代价**：开发量大于 CLI，需处理 STT/TTS 管线与 WebSocket 流式。

## ADR-002：一切外部依赖皆适配器（存储 + 模型 + 语音）

- **决策**：存储、LLM、STT、TTS、发音评估全部用「接口 + 适配器」抽象。
- **理由**：用户要求支持云服务商模型与自部署模型；存储要可在本地/云端间选择。统一成适配器模式，业务代码零耦合。
- **落地**：LLM 重点做 `OpenAICompatAdapter`（覆盖 80% 模型）+ `ClaudeAdapter`（原生，做评分）。

## ADR-003：发音评估不自研，留适配器，默认关闭

- **决策**：不自研发音评估；做成 `PronunciationProvider` 适配器，第一版默认 `NoneAdapter`。
- **理由**：发音评估是独立的声学难题，自研等于重做声学模型，投入产出极差；成熟方案（Azure Pronunciation Assessment、讯飞、Kaldi GOP）已很好。
- **第一版行为**：口语打分中发音(pronunciation)、流利度(fluency)维度标注「基于文本估算，仅供参考」，其余维度（内容/词汇/语法/连贯）正常评分。用户填 Azure key 即升级为音素级评分。

## ADR-004：生词只存来源句，不存释义

- **决策**：`VocabEntry` 存 `word + lemma + context_sentences[]`，**不存任何释义文字**。
- **理由**：
  - 只存单词会丢失语境，回头不记得为何不认识，违背"理解式记忆"。
  - 存释义会：① 多一次 LLM 成本；② 制造"标准答案"，用户去比对 → 滑回死记硬背；③ 多义词存哪个义项是麻烦。
  - 只存来源句：空间极省（句子本就在手）、语境天然消歧（"sat by the river bank"自锁定河岸义）、完美契合功能3a（意思由用户当场理解出来）。
- **一句话**：意思不该被存储，意思应在复习时被重新理解出来；来源句是钥匙。
- **多义处理**：同词不同义 → 存多条 `context_sentences`，背诵时一起呈现，建立立体认知。

## ADR-005：考试模式零脚手架，但允许提前交卷

- **决策**：考试模式（free_write / dialogue）不救场、不给提示，但用户可随时「提前交卷」，按已有内容打分+复盘。
- **理由**：压力是刻意设计，防止用户形成依赖；提前交卷不是救场（不给词/提示），只是允许早结束，符合真实考试。
- **附带**：卡顿/停顿作为流利度评分信号记录进复盘，不抹平。

## ADR-006：模型按任务分配

- **决策**：评分用最强模型，切词等高频轻任务用本地/轻量模型，对话用性价比模型。用户可配置。
- **理由**：兼顾评分准确性与高频任务成本。

## ADR-007：无账号系统，单用户本地优先，schema 预留 user_id

- **决策**：不做注册/登录；默认单用户（`user_id = "local-user"`），本地优先。但数据模型保留 `user_id` 字段。
- **理由**：开源自部署工具的典型用户是 clone 下来本地自用（参考 Anki 等）。强加账号是负担，劝退使用者与贡献者。保留 `user_id` 字段则让未来 fork 成多用户 SaaS 只需加一层认证，零成本保留可能性。

## ADR-008：后端语言定 Python (FastAPI)

- **决策**：后端用 Python + FastAPI；前端 React/TS。
- **理由**：核心依赖几乎全在 Python 生态 —— spaCy（切词/词元还原）、faster-whisper（本地 STT）、FSRS（官方库）。用 Node 会被迫起 Python 子进程做切词和语音，变成双语言维护。不为"前后端同语言"牺牲整个 ML 生态。

## ADR-009：开源落地基线纳入 MVP

- **决策**：数据导入/导出、首次配置向导、一键启动（docker-compose）、「今日学习」首页 一并纳入 MVP，而非阶段2。
- **理由**：开源项目"能不能被人跑起来、能不能形成日常习惯"由这几项直接决定。
  - 导入/导出（尤其生词本导出 Anki）：本地优先用户最怕数据被锁死；Anki 兼容是引流与留存点。
  - 配置向导：落地第一道坎是配置 API key 与模型，无向导则新用户跑不起来。
  - 「今日学习」首页：三大功能是平行的，需要一个聚合页把碎片串成日常习惯，否则用户不知从何下手。

## ADR-010：多义词调度单位 = 按词（per-lemma），不按义项

- **决策**：FSRS 调度以 `VocabEntry`（即一个 lemma）为单位，一个生词条目持有**唯一** `fsrs_state`；同词不同义共用同一调度节奏。多义靠 `context_sentences` 多条并呈消解，不拆成多个义项各自调度。
- **背景**：这是 `07-implementation-order.md` 列出的红线——「一个 VocabEntry 存多条 context_sentences（多义）却只有一个 fsrs_state，按词还是按义项调度，需在 L2 接 FSRS 前定下」。L2 接入 FSRS 前在此定案。
- **理由**：
  - schema 已对 `(user_id, lemma)` 建唯一索引、每个 `VocabEntry` 仅一个 `fsrs_state`——按义项需为每个义项存独立调度状态，与既有结构冲突。
  - 按义项必须为每个义项贴标签/释义以区分，等于变相存释义，违背 ADR-004（不存释义）。
  - 理解式记忆（功能3a）本就把多义来源句**一起**呈现以建立「立体认知」；同词多义一起复习正是该设计，不该被拆散。
  - 简单可解释：用户看到的是「这个词到期了」，而非「这个词的第2个义项到期了」。
- **代价**：某词一个义项熟、另一义项生时，调度无法分别对待（整体一个节奏）。可接受——多义来源句并呈时熟义项顺带复习成本极低；真有强需求留待阶段2的「义项级」扩展（需新 ADR + schema 迁移）。
- **关联**：ADR-004（只存来源句不存释义）；落地见 `app.scheduling.fsrs_scheduler` 与 `FsrsState`。
