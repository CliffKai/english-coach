# 功能实现依赖顺序

> 本文回答「先写什么、后写什么」。`06-roadmap.md` 给的是范围与进度；本文给的是**严格的依赖拓扑**——上游没好，下游写不了。
> 原则：先地基（适配器接口 + 存储），再单点功能，最后串成日常闭环。每完成一层应可独立验证。

## 依赖总图

```
                    ┌──────────────────────────────────┐
            L0 地基  │ 适配器接口骨架 + 配置加载 + DB schema │
                    └───────────────┬──────────────────┘
                          ┌─────────┴─────────┐
            L1 适配器实现  │ LLMProvider   存储Repo │  STT/TTS(L4再接)
                    └─────────┬─────────┘
            L2 NLP/调度  │ spaCy切词·lemma   FSRS调度器 │
                    └─────────┬─────────┘
   ┌──────────────┬──────────┴──────────┬───────────────┐
L3 │ F1 生词收集   │  水平基线分级         │               │
   └──────┬───────┴──────────┬──────────┘               │
          │                  │                          │
L3 │ F3a 理解式背词 │  F2c 自由写作打分 → ErrorAnalysis → 错题本 │
   └──────┬─────────────────┬────────────────────────────┘
          └────────┬────────┘
            L4 语音  │ STT/TTS 接入 → F2d 对话 · F2a/2b 引导 · F3b 造句 │
                    └────────┬────────┘
            L5 闭环  │ 「今日学习」首页  ·  导入/导出(Anki)  ·  配置向导收尾 │
                    └─────────────────────────────────────────┘
```

## 逐层说明

### L0 — 地基（无依赖，最先做）
- **适配器接口定义**：`LLMProvider` / `WordRepository` / `ErrorRepository` / `SessionRepository` / `STTProvider` / `TTSProvider` / `PronunciationProvider`，只定接口与数据类型，不写实现。
- **配置加载**：读 `Settings`（含 per-task `model_config`），`.env.example` 落地。
- **DB schema / DDL**：`VocabEntry` / `ErrorEntry` / `PracticeSession` / `Settings` 四表，所有表带 `user_id`（默认 `local-user`，ADR-007）。
- **前后端脚手架**：FastAPI app + React/TS 空壳能起来。
- ✅ 验证：服务能启动，接口可被 mock 注入。

### L1 — 适配器实现（依赖 L0 接口）
- `LLMProvider`：先 `OpenAICompatAdapter`（覆盖 ~80% 模型）+ `ClaudeAdapter`（评分用）。
- 存储：`LocalAdapter`（SQLite）实现三个 Repo。
- ⚠️ `PronunciationProvider` 此层只做 `NoneAdapter`（ADR-003）。STT/TTS 推迟到 L4。
- ✅ 验证：能存取一条 VocabEntry；能用配置的模型跑通一次 chat。

### L2 — NLP 与调度（依赖 L1 存储）
- **spaCy 切词 + lemma 还原 + 词频过滤**（F1 和水平基线都要用，必须前置）。
- **FSRS 调度器**：给定 `fsrs_state` 算「今日该复习哪些词」（F3 的引擎）。
- ✅ 验证：一段英文 → 词元列表；一批词 → 到期复习队列。

### L3 — 核心学习闭环（本项目的价值核心）
**严格顺序**——后者吃前者的产物：

1. **水平基线分级**：首次分级（CEFR / 估算雅思分），写入 `Settings.level_baseline`。F1 过滤与 F2 打分都参照它，故先做。
2. **F1 生词收集**：切词 → 按基线/词频过滤 → 逐词问询 → 不认识者连同来源句入库。**产出 VocabEntry**（喂给 F3）。
3. **F3a 理解式背词**：消费 F1 的生词 + L2 的 FSRS 队列；展示来源句让用户复述理解（不比对标准释义，ADR-004）。
4. **F2c 自由写作打分**：考试模式，延迟纠错 → 雅思/托福多维度打分。**产出 ErrorEntry**。
5. **ErrorAnalysisAgent + 错题本**：汇总 F2c 的隐藏 buffer，产复盘、回填错题本。**必须紧跟 F2c**（消费其产物）。

> F1→F3a 是「生词河」，F2c→ErrorAnalysis 是「错误河」，两河此层即贯通。
- ✅ 验证：粘贴文本能攒出生词并背诵；写一段文章能拿到分数 + 错题。

### L4 — 语音（依赖 L1 接口 + L3 的练习逻辑）
- 先接 `STTProvider`（faster-whisper）+ `TTSProvider`。
- 然后依赖语音的功能：**F2d 对话打分**（语音对话 + 延迟纠错，复用 ExaminerAgent）、**F2a/2b 引导写/说**（练习模式即时纠错，TutorAgent）、**F3b 语境造句背**。
- ⚠️ F2d 强依赖 STT/TTS，故整组放语音之后；F2a/b 的写作分支理论上可早做，但归并到此层统一处理纠错 UI。
- ✅ 验证：能录音转写、合成播放，跑通一次口语对话打分。

### L5 — 日常闭环 / 开源落地（依赖前面所有功能存在）
- **「今日学习」聚合首页**：待复习生词(FSRS) + 待巩固错题(ErrorEntry) + 推荐话题。**必须等 F1/F3/F2 都有数据产出后才有意义**。
- **导入/导出**：JSON 全量备份 + 生词本导出 Anki（CSV/.apkg）。
- **配置向导收尾**：把 L1 的模型配置、L3 的水平基线测试串成首次引导流程；docker-compose 一键启动。
- ✅ 验证：新用户 clone → 一键起 → 向导走完 → 首页告诉他「今天学什么」。

## 关键依赖红线（写代码前牢记）
- **接口先于实现**：L0 的接口不定好，L1 之后全部返工。
- **水平基线先于 F1/F2**：过滤和打分都读它。
- **F1 先于 F3**：F3 没生词可背。
- **F2c 紧贴 ErrorAnalysis**：错误 buffer 是临时的，产出即消费。
- **STT/TTS 先于 F2d**：对话打分无语音不成立。
- **闭环页/导出最后做**：它们聚合上游数据，上游空则页面空。

## 已知风险（走到对应层时必须正面解决，不能靠排序绕开）
依赖顺序只解决「能不能现在做」，不解决「做出来对不对」。以下几堵墙照顺序做也躲不掉，到对应层时务必处理：

- **L3 / 打分可信度**：LLM 对作文/口语评分方差大、难校准。需要固定 rubric + 锚定样例 + 一致性测试（同篇两次分数应稳定），并明确标注「AI 估算」。无 Azure 发音评估时口语分如何取信（ADR-003 标注「基于文本估算」）。
- ~~**L3 / FSRS 评级映射**：F3a 是开放式「说出理解」由 LLM 判断，但 FSRS 要离散评级（again/hard/good/easy）。这个「模糊判断 → 离散评级」的映射必须先定义清楚，否则调度全错。~~ **已定：LLM 只输出 correct/partial/wrong（+用户自评 too_easy），确定性映射到 good/hard/again（+easy），见 ADR-011。** 落地于 `MemoryWordAgent`。
- **L2-L3 / 多义词调度单位**：~~一个 `VocabEntry` 存多条 `context_sentences`（多义）却只有一个 `fsrs_state`。按词还是按义项调度，需在 L2 接 FSRS 前定下。~~ **已定：按词调度（per-lemma），见 ADR-010。** 同词多义共用一个 `fsrs_state`，多义来源句一起复习。
- **L5 / 不存释义 vs Anki 导出**：ADR-004 不存释义，但 Anki 卡片要正反面 —— 卡背放什么？做导出前要想清楚（来源句？现场生成释义？）。
- **贯穿 / Agent 评测**：五个 Agent 全 prompt 驱动，改 prompt 容易按下葫芦浮起瓢。至少给切词判断、错误检测、评分各备一组黄金样例做回归。

## 与 roadmap 的对应
本文 L2–L3 ≈ roadmap `1b`，L4 ≈ `1c`，L5 ≈ `1d`。L0–L1 主要对应 `1a`（脚手架/存储/LLM 适配器）；注意 `1a` 里的「配置向导」本文拆成两半 —— **接口/模型配置在 L1，向导 UI 收尾在 L5**（要等水平基线等流程都在了才能串成首次引导）。roadmap 管「做不做、做没做」，本文管「能不能现在做」。
